## MiniArray

MiniArray 是每个 Cubby 里的变长元数据数组，用来保存每个 slot 对应的 MetaEntry 编码结果。它的外部接口很小：reset 初始化固定 entry 数，update 更新某个 entry，access 读取某个 entry，bits_total 统计逻辑 bit 数。

```cpp
class MiniArray {
public:
  using Bits = std::vector<uint64_t>;

  MiniArray() = default;
  explicit MiniArray(size_t n) { reset(n); }

  void reset(size_t n) {
    bitlens_.assign(n, 0);
    offsets_.assign(n + 1, 0);
    entries_.assign(n, Bits{});
    buf_.clear();
  }
};
```

内部有四份状态：

- bitlens_：每个 entry 的 bit 长度。
- offsets_：前缀和，offsets_[i] 是第 i 个 entry 在 packed buffer 里的起始 bit。
- entries_：每个 entry 的原始 Bits 副本。
- buf_：真正连续打包后的 bit buffer。

更新时不是局部改 buffer，而是先保存新 entry，再整体 repack()：

```cpp
void update(size_t i, const Bits &bits, uint32_t bitlen) {
    if (i >= bitlens_.size())
      throw std::out_of_range("MiniArray::update");
    bitlens_[i] = bitlen;
    entries_[i] = bits;
    repack();
}
```

repack() 会重新算所有偏移，然后按 bit 把 entries_ 拷贝进 buf_：

```cpp
void repack() {
    recompute_offsets();
    const uint64_t total = offsets_.back();
    buf_.assign(static_cast<size_t>((total + 63) / 64), 0);
    
    for (size_t i = 0; i < bitlens_.size(); i++) {
      const uint32_t bl = bitlens_[i];
      if (bl == 0)
        continue;
      const uint64_t start = offsets_[i];
      for (uint32_t b = 0; b < bl; b++) {
        set_bit(start + b, get_bit_from(entries_[i], b));
    }
}
```

一个值得注意的点：虽然 buf_ 是紧凑打包的，但类里还保留了 entries_ 原始副本，所以真实内存并不完全等于 bits_total()；bits_total() 更像逻辑空间/指标口径。

## Router

Router 是 facility 内的本地查询路由。它把完整 key 映射到 (Cubby*, slot)，即某个元素实际在哪个小隔间和槽位。

```cpp
class Router {
public:
  using Value = std::pair<Cubby *, size_t>;

  std::pair<std::optional<Value>, uint64_t> locate(uint64_t key) const;
  bool contains(uint64_t key) const;
  uint64_t insert(uint64_t key, Value v);
  uint64_t erase(uint64_t key);
  void clear();
  // 编码后的 bitstream 大小（用于空间统计）。
  uint64_t bits_total() const { return static_cast<uint64_t>(encoded_bits_); 
}
```

它的核心结构是一个基于 64-bit hash 串的 Patricia/binary trie：

```cpp
  struct Entry {
      uint64_t key = 0;
      Value v{};
      uint64_t bits = 0; // hashed bits used by trie
    };
  
  struct Node {
      int left = -1;
      int right = -1;
      int entry_idx = -1;    // leaf: index into entries_
      int bucket_idx = -1;   // hash-collision bucket (split_bit==64)
      uint8_t split_bit = 0; // internal node: branching bit position [0,64]
  };
  
  std::vector<Entry> entries_;
  std::vector<Node> nodes_;
  std::vector<std::vector<int>> buckets_;
```

key 不直接按原值建树，而是通过 splitmix64 得到稳定随机 bit 串：

```cpp
  static uint64_t key_bits(uint64_t key) {
    // 与设计文档一致：router 对键使用随机 bit 串；这里用 splitmix64 作为稳定hash。
    return splitmix64(key ^ 0x9e3779b97f4a7c15ULL);
  }
```

插入和删除都很直接：更新 entries_，然后整体 rebuild()。

```cpp
uint64_t Router::insert(uint64_t key, Value v) {
  // overwrite semantics
  for (auto &e : entries_) {
    if (e.key == key) {
      e.v = v;
      rebuild();
      return 1;
    }
  }
  entries_.push_back(Entry{key, v, key_bits(key)});
  rebuild();
  return 1;
}

uint64_t Router::erase(uint64_t key) {
  auto it = std::remove_if(entries_.begin(), entries_.end(),
                           [&](const Entry &e) { return e.key == key; });
```

rebuild() 的流程是：

1. 清空 nodes_ / buckets_。
2. 重新计算所有 entry 的 hash bits。
3. 构造 entry index 数组。
4. 调 build_range() 递归建 Patricia trie。
5. 调 recompute_encoded_bits() 计算路由结构的编码位数。

查询 locate() 则沿 trie 的 split_bit 判断左右子树，遇到普通叶子直接比 key，遇到 bucket 则线性扫 bucket：

```cpp
std::pair<std::optional<Router::Value>, uint64_t> Router::locate(uint64_t key) const {
  // steps：访问 trie 的节点数（Patricia 深度 <= 64）。
  uint64_t steps = 0;
  uint64_t bits = key_bits(key);
  int node = root_;
  while (node >= 0 && node < static_cast<int>(nodes_.size())) {
    steps++;
    const Node &nd = nodes_[node];
    if (nd.bucket_idx >= 0) {
      const auto &bucket = buckets_[static_cast<size_t>(nd.bucket_idx)];
      for (int idx : bucket) {
        steps++;
        const Entry &e = entries_[static_cast<size_t>(idx)];
        if (e.key == key)
          return {e.v, steps};
      }
      return {std::nullopt, steps};
    }
```

bits_total() 不是保存真实序列化 buffer，而是通过局部 BitWriter 走一遍树，计算如果编码要占多少 bit。编码规则大致是：

- internal node：1 bit is_leaf + 6 bits split_bit + left + right
- normal leaf：1 bit is_leaf + 1 bit bucket_flag + 64 bits key
- bucket leaf：1 bit is_leaf + 1 bit bucket_flag + 8 bits count + count * 64 bits key

两者在系统里的位置

Facility 里有一组 Router：vector `<Router>` D;

Cubby 里有一个 MiniArray：

```cpp
  // 商压缩：存 quotient payload（π(x) 的高位）；完整键由 Meta + 路由上下文恢复。
  std::vector<std::optional<uint64_t>> slots;
  std::vector<size_t> occupied;               // 占用索引数组

  FreeSlotTree free_slots; // 设计文档：快速找空槽
  MiniArray meta;          // 元数据
```

插入时，Router 记录 “key -> cubby/slot”，MiniArray 记录该 slot 的恢复元信息：

```cpp
  void put_quotient_slot(Cubby &c, size_t slot, uint64_t key, uint64_t gx_pi,
                         size_t facility_r, const TableState &t, uint32_t ins) {
    (void)facility_r;
    const uint32_t ln = logn_of(t);
    const MetaCodecLayout ly = meta_layout(t.K, c.capacity);
    const MetaEntry me = make_meta_entry(key, gx_pi, slot, facility_r, t.N, t.K,
                                         c.capacity, ins);
    auto pr = encode_meta_entry(me, ly);
    c.slots[slot] = quotient_payload(gx_pi, ln);
    c.meta.update(slot, pr.first, pr.second);
```

查询时先用 Router 找到候选 (cubby, slot)，再用 MiniArray 里的 meta 和 slot 里的 quotient payload 重建 π(key)，确认是否匹配：

```cpp
  bool slot_pi_matches(const Cubby &c, size_t slot, size_t facility_r,
                       const TableState &t, uint64_t key_gx) const {
    if (slot >= c.slots.size() || !c.slots[slot].has_value())
      return false;
    MetaEntry me;
    if (!decode_slot_meta(c, slot, t, me))
      return false;
    const auto gx = reconstruct_gx_pi(facility_r, t.N, t.K, c.capacity, slot,
                                      *c.slots[slot], me, logn_of(t));
    return gx.has_value() && *gx == key_gx;
```

总结

MiniArray 负责 cubby 内“每个槽的变长元数据压缩存储”，当前实现简单可靠，但更新会整体重打包，且保留原始 entries_ 副本。

Router 负责 facility 内“从 key 快速定位到 cubby/slot”，当前实现用 64-bit hash Patricia trie，插入/删除后整体重建，查询路径最多约 64 层，hash 碰撞时走 bucket 线性查找。

整体上，这两个类都属于论文结构的工程简化版：优先保证功能、指标可观测和实现清晰，而不是严格达到论文里的常数时间动态更新与最优紧凑空间

## k-kick 树

```cpp
// 论文 k-kick 树：当前由 HashTable::Impl::kkick_insert（src/ht.cpp）在 tail 单 cubby
// 内有界随机探针 + 踢键链实现；本类仅保留 k 参数语义占位，供后续对齐完整树结构。
class KKickTree {
public:
  explicit KKickTree(int k) : k_(k) {}
  int k() const { return k_; } // 层数
```

真正的“k-kick 行为”在 src/ht.cpp 的 HashTable::Impl::kkick_insert() 里，是一个工程简化版：在单个 Cubby 内做有界随机探针和踢出链，而不是论文里的完整多层 k-kick tree。

插入主流程里，先通过 Router 检查 key 是否已存在，然后确保 facility 有一个未满 tail cubby，最后把 key 插入 tail：

```cpp
ensure_tail(active_, f);

// Step D: k-kick insertion into tail cubby
auto kr = kkick_insert(f, f.tail, key,
                       facility_index_of(active_, &f));
```

ensure_tail() 的作用是：如果当前 tail 还有空间就复用；如果 tail 满了，就把旧 tail 挂回 tiers[tail_tier]，再新建一个 tier 0 的 tail cubby。

```cpp
void ensure_tail(TableState &t, Facility &f) {
    if (f.tail && f.tail->size < f.tail->capacity)
      return;
// 如果旧 tail 已满，把它作为“满 cubby”挂回 tiers[tail_tier]。
    if (f.tail_owned && f.tail_owned->size == f.tail_owned->capacity) {
      if (static_cast<int>(f.tiers.size()) <= f.tail_tier)
        f.tiers.resize(static_cast<size_t>(f.tail_tier + 1));
      f.tiers[static_cast<size_t>(f.tail_tier)].push_back(
          std::move(f.tail_owned));
      f.tail = nullptr;
}
```

kkick_insert() 的核心逻辑是：

1. cur_key 初始为要插入的新 key。
2. 对 step = 0 .. kmax 做探针，kmax 来自 params_.k。
3. 如果探到空槽，直接写入，更新 MiniArray、FreeSlotTree 和 Router，结束。
4. 如果探到已占用槽，恢复该槽里的旧 key，把旧 key 从 Router 删除，把当前 key 写入这个槽，再把被踢出的 key 变成新的 cur_key 继续下一轮。
5. 如果超过 k 次还没遇到空槽，则用 FreeSlotTree 找任意空槽兜底放入。

   ```cpp
   const uint64_t kmax = static_cast<uint64_t>(std::max(0, params_.k));
   uint64_t cur_key = key;

   auto router_erase_key = [&](uint64_t ky) {
     router_at(f, route_bucket_for(active_, pi_.pi(ky))).erase(ky);
   };
   auto router_put_key = [&](uint64_t ky, Cubby *cb, size_t slot) {
     router_at(f, route_bucket_for(active_, pi_.pi(ky)))
         .insert(ky, std::make_pair(cb, slot));
   };

   for (uint64_t step = 0; step <= kmax; step++) {
     const size_t pos = probe_slot(cur_key, c->capacity, step);
     if (!c->slots[pos].has_value()) {
       c->occupied.push_back(pos);
       c->size++;
       const uint64_t gxc = pi_.pi(cur_key);
       put_quotient_slot(*c, pos, cur_key, gxc, facility_r, active_,
                         static_cast<uint32_t>(step));
       c->free_slots.mark_used(static_cast<int>(pos));
       router_put_key(cur_key, c, pos);
       out.ok = true;
       out.slot = pos;
       out.moved = step;
       return out;
     }
   ```

探针位置由 probe_slot() 生成：先对 key 做可逆置换 pi(key)，再和 step salt 混合后 splitmix64，最后对 cubby 容量取模。

```cpp
  size_t probe_slot(uint64_t key, size_t cap, uint64_t i) const {
    // 伪随机探针序列：splitmix64(key_bits ^ i) % cap
    // 先用 pi(key) 打散，避免 key 低位模式影响。
    const uint64_t gx = pi_.pi(key);
    const uint64_t h = splitmix64(gx ^ (i * 0x9e3779b97f4a7c15ULL));
    return static_cast<size_t>(cap == 0 ? 0 : (h % cap));
  }
```

如果发生踢出，代码会先从槽内的 quotient payload + MiniArray meta 恢复被踢 key，然后维护 Router 的一致性：

```cpp
  const auto kicked_opt =
      recover_key_from_slot(*c, pos, facility_r, active_);
  if (!kicked_opt) {
    out.ok = false;
    return out;
  }
  const uint64_t kicked_key = *kicked_opt;

  router_erase_key(kicked_key);
  const uint64_t gxc = pi_.pi(cur_key);
  put_quotient_slot(*c, pos, cur_key, gxc, facility_r, active_,
                    static_cast<uint32_t>(step));
  router_put_key(cur_key, c, pos);
  cur_key = kicked_key;
```

所以这里的 kick_count 实际就是 kkick_insert() 返回的 moved，用于指标记录。TableParams::k 默认是 2，注释也说明它是对接设计文档 k-kick 层数参数：

```cpp
struct TableParams {
  uint64_t n = 10000;        // 规模参数（N 会取 2^p，使 N <= n <= 2N）
  int k = 2;                 // 保留（对接设计文档的 k-kick 层数参数）
  double load_factor = 0.90; // 保留
```

总结一下：当前 KKickTree 不是一个真正的数据结构实现，而是“k 参数 + bounded kick chain”的工程落地。完整论文版 k-kick tree 强调多层 cubby、分层饱和状态和跨层受控驱逐；本项目当前实现只在单个 cubby 内做最多 k + 1 次探针/踢出，失败后用空闲槽树兜底。这能支撑功能正确性和 kick_count 指标观测，但不等价于论文里的严格渐近结构。

下面按当前代码实际执行梳理。核心入口都在 src/ht.cpp 的 HashTable::Impl。

## 地址拆分与路由

所有操作先算：

```cpp
gx = pi_.pi(key)
```

pi_ 是 3 轮 Feistel 可逆置换，所以后面可以通过 pi_.inverse(gx) 恢复原 key。

gx 的低 logN 位叫 g*：

```cpp
  static size_t route_bucket_for(const TableState &t, uint64_t gx) {
    if (t.N < 2 || t.K == 0)
      return 0;
    const uint64_t g = gx & (t.N - 1);
    return static_cast<size_t>(g % t.K);
```

对应关系是：

- g = gx & (N - 1)：取 π(key) 的低 logN 位。
- facility_r = g / K：选择哪个 Facility。
- bucket_b = g % K：选择该 facility 的哪个 Router D[b]。
- Router 再定位到具体 (Cubby*, slot)。

### 插入过程

入口是 insert()，加锁后调用 insert_no_lock()；成功插入后可能触发扩缩容、后台 rebuild、渐进迁移。

```cpp
  InsertResult insert(uint64_t key) {
    std::lock_guard<std::mutex> lk(mu_);
    auto r = insert_no_lock(key, true);
    if (r.ok && r.inserted)
      maybe_resize_locked();
    if (r.ok)
      run_rebuild_budget();
    run_migrate_budget();
    return r;
  }
```

insert_no_lock() 主流程：

1. 检查是否初始化。
2. 计算 gx = pi_.pi(key)。
3. 找 active 表中的 facility 和 router bucket。
4. 先查 active router，如果 key 已存在，直接返回 inserted=false。
5. 如果正在迁移，还有 old_，再查 old 表，存在也返回 inserted=false。
6. 确保当前 facility 有可用 tail cubby。
7. 调 kkick_insert() 插入 tail。
8. 更新 n_、指标、可能调度 rebuild。

   ```cpp
   uint64_t gx = pi_.pi(key);
   Facility &f = facility_for_key(active_, gx);
   const size_t b = route_bucket_for(active_, gx);
   if (auto loc = router_at(f, b).locate(key); loc.first.has_value()) {
     r.ok = true;
     r.inserted = false;
     r.router_probe_steps = loc.second;
     r.kick_count = 0;
     r.cubby_tier = -1;
     const uint64_t meta_bits = facility_router_bits_sum(f);
     global_metrics().on_insert(/*moved=*/0, /*router_steps=*/loc.second,
                                /*meta_bits=*/meta_bits);
     return r;
   }
   // ... old_ 查重 ...
   ensure_tail(active_, f);

   // Step D: k-kick insertion into tail cubby
   auto kr = kkick_insert(f, f.tail, key,
                          facility_index_of(active_, &f));
   ```

kkick_insert() 是当前工程版 k-kick：在单个 cubby 内最多探测/踢出 k 轮。每次探针位置由 probe_slot(cur_key, capacity, step) 给出。

如果探到空槽：

- put_quotient_slot() 写入商压缩 payload 和 meta。
- free_slots 标记占用。
- Router D[b] 插入 key -> (cubby, slot)。

如果探到占用槽：

- 用 recover_key_from_slot() 恢复被踢出的旧 key。
- 从 Router 删除旧 key。
- 把当前 cur_key 写入该 slot。
- Router 改为登记当前 key。
- cur_key = kicked_key，继续下一轮。

超过 k 后还没空槽，则用 FreeSlotTree 找任意空槽兜底。

### 查询过程

查询只读，不改结构：

1. 未初始化返回错误。
2. 算 gx，定位 active facility 和 bucket。
3. 用 Router::locate(key) 找候选 (Cubby*, slot)。
4. 不直接信 Router，继续用 slot_pi_matches() 验证 slot 内压缩信息能否重建出同一个 gx。
5. 如果 active 未找到且存在 old_，同样查 old 表。

   ```cpp
   uint64_t gx = pi_.pi(key);
   const Facility &f = facility_for_key(active_, gx);
   r.ok = true;
   const size_t b = route_bucket_for(active_, gx);
   auto [loc, steps] = router_at(f, b).locate(key);
   r.router_probe_steps = steps;
   r.found = loc.has_value() &&
             slot_pi_matches(*loc->first, loc->second,
                              facility_index_of(active_, &f), active_, gx);
   // ...
   if (!r.found && old_) {
     const Facility &of = facility_for_key(*old_, gx);
     const size_t ob = route_bucket_for(*old_, gx);
     auto [oloc, osteps] = router_at(of, ob).locate(key);
   ```

slot_pi_matches() 做的就是：从 MiniArray 解出 meta，再结合 slot 中的 payload 重建 gx，与查询 key 的 pi(key) 比较。

### 删除过程

删除也先查 Router：

1. 算 gx，定位 active facility/bucket。
2. Router::locate(key) 找 slot。
3. 找到则清空 slot：

   - slots[slot].reset()
   - meta.update(slot, {}, 0)
   - 从 occupied 删除
   - free_slots.mark_free()
   - Router.erase(key)
   - n_--
4. 如果删除的不是 tail cubby，则从 tail 搬一个元素来填洞，保持只有 tail 允许不满：

   - 从 tail 最后一个 occupied slot 恢复 key。
   - 清空 tail 旧位置。
   - 用 put_quotient_slot() 写入被删除位置。
   - 更新 Router 指向新位置。
5. 如果存在 old_，也在 old 表尝试删除同一个 key。
6. 记录指标，可能触发 resize、rebuild、migration。

   if (loc) {

## 商压缩逻辑

slot 里不直接存完整 key，也不直接存完整 gx = π(key)。当前存两部分：

- slots[slot]：payload = gx >> logN，也就是去掉低 logN 位后的高位商。
- MiniArray meta[slot]：恢复低 logN 位所需的辅助信息。

写入发生在 put_quotient_slot()：

```cpp
  void put_quotient_slot(Cubby &c, size_t slot, uint64_t key, uint64_t gx_pi,
                         size_t facility_r, const TableState &t, uint32_t ins) {
    (void)facility_r;
    const uint32_t ln = logn_of(t);
    const MetaCodecLayout ly = meta_layout(t.K, c.capacity);
    const MetaEntry me = make_meta_entry(key, gx_pi, slot, facility_r, t.N, t.K,
                                         c.capacity, ins);
    auto pr = encode_meta_entry(me, ly);
    c.slots[slot] = quotient_payload(gx_pi, ln);
    c.meta.update(slot, pr.first, pr.second);
  }
uint64_t quotient_payload(uint64_t gx_pi, uint32_t logN) {
  if (logN >= 64)
    return 0;
  return gx_pi >> logN;
}
```

MetaEntry 里关键字段：

- delta = g_i - slot_index
- mid_bits = g_k / cubby_capacity
- router_bits：16 bit 指纹，目前编码了但查询恢复逻辑没有使用。
- insert_bits：记录插入/kick step，8 bit。

```cpp
MetaEntry make_meta_entry(uint64_t key, uint64_t gx_pi, size_t slot_index,
                          uint64_t facility_r, uint64_t N, uint64_t K,
                          size_t cubby_capacity, uint32_t ins_tag) {
  MetaEntry e;
  (void)facility_r;
  (void)N;
  const size_t cap = std::max<size_t>(1, cubby_capacity);
  const uint64_t g_star = gx_pi & (N - 1);
  const uint64_t g_k = g_star & (K - 1);
  const uint64_t g_i = g_k % cap;
  e.delta = static_cast<int64_t>(g_i) - static_cast<int64_t>(slot_index);
}
```

这里依赖 K 是 2 的幂，所以 g_star & (K - 1) 等价于 g_star % K。choose_K() 确实会把 K 选成 2 的幂。

### 恢复键逻辑

恢复分两步：先恢复 gx = π(key)，再用 Feistel 逆置换恢复原 key。

```cpp
recover_key_from_slot()：

  std::optional<uint64_t> recover_key_from_slot(const Cubby &c, size_t slot,
                                                size_t facility_r,
                                                const TableState &t) const {
    if (slot >= c.slots.size() || !c.slots[slot].has_value())
      return std::nullopt;
    MetaEntry me;
    if (!decode_slot_meta(c, slot, t, me))
      return std::nullopt;
    const auto gx = reconstruct_gx_pi(facility_r, t.N, t.K, c.capacity, slot,
                                        *c.slots[slot], me, logn_of(t));
    if (!gx.has_value())
      return std::nullopt;
    return pi_.inverse(*gx);
  }
```

reconstruct_gx_pi() 的数学过程：

1. g_i = slot_index + delta
2. g_k = mid_bits * cubby_capacity + g_i
3. g_star = facility_r * K + g_k
4. gx = (payload << logN) | g_star

```cpp
std::optional<uint64_t> reconstruct_gx_pi(uint64_t facility_r, uint64_t N,
                                           uint64_t K, size_t cubby_capacity,
                                           size_t slot_index, uint64_t payload,
                                           const MetaEntry &m, uint32_t logN) {
  if (N < 2 || K == 0 || logN == 0 || logN >= 64)
    return std::nullopt;
  const size_t cap = std::max<size_t>(1, cubby_capacity);
  const int64_t g_i = static_cast<int64_t>(slot_index) + m.delta;
  if (g_i < 0 || static_cast<uint64_t>(g_i) >= cap)
    return std::nullopt;
  const uint64_t g_k =
      static_cast<uint64_t>(m.mid_bits) * cap + static_cast<uint64_t>(g_i);
```

所以完整恢复链是：

```text
slot payload + MiniArray meta + facility_r + slot_index
=> reconstruct_gx_pi(...)
=> gx = π(key)
=> pi_.inverse(gx)
=> key
```

## 扩容缩容

当前实现的要点是：Router 负责快速找到候选位置，商压缩数据负责验证和恢复真实 key；

当前扩缩容不是单次全量 rehash，而是“双表渐进迁移”：触发 resize 后，把旧 active_ 挪到 old_，新建一个空的 active_，之后每次插入/删除顺手迁移一个旧 facility。

### 触发条件

扩缩容入口是 maybe_resize_locked()：

```cpp
  void maybe_resize_locked() {
    if (active_.facilities.empty() || active_.N < 2)
      return;
    const double lf = (params_.load_factor > 0.0 && params_.load_factor < 1.0)
                          ? params_.load_factor
                          : 0.90;
    if (!old_ && static_cast<double>(n_) > static_cast<double>(active_.N) * lf) {
      start_migration(active_.N * 2);
    } else if (!old_ && n_ > 0 && n_ < active_.N / 4) {
      start_migration(std::max<uint64_t>(2, active_.N / 2));
    }
  }
```

也就是说：

- 扩容：n_ > active_.N * load_factor，默认负载因子是 0.90。
- 缩容：n_ > 0 && n_ < active_.N / 4。
- 如果已经有 old_ 正在迁移，则不会启动新的扩缩容。
- 新容量会经过 next_pow2()，所以 N 始终是 2 的幂。

insert() 只有真正插入成功后才检查 resize；erase() 删除后也会检查 resize。查询不会触发 resize，也不会推进迁移。

### 开始迁移

start_migration(newN) 做四件事：

1. 清空 rebuild scheduler。
2. 记录 resize start 指标。
3. old_ = std::move(active_)，旧表整体变成 old。
4. 重建一个新的空 active_，包含新的 N、K、facilities、routers 和 tail cubbies。

```cpp
  void start_migration(uint64_t newN) {
    // Phase7: 双表迁移（渐进）
    scheduler_.clear();
    global_metrics().on_resize_start();
    old_ = std::move(active_);
    active_.reset();
    migrate_progress_ = 0;
    active_.N = next_pow2(newN);
    active_.K = choose_K(active_.N);
    const uint64_t facilities_cnt = std::max<uint64_t>(1, active_.N / active_.K);
    active_.facilities.resize(static_cast<size_t>(facilities_cnt));
    for (auto &f : active_.facilities) {
      f.tiers.clear();
      f.max_tier = std::max(0, params_.k);
      f.D.assign(static_cast<size_t>(active_.K), Router{});
```

这里 n_ 不会清零，因为元素总数没变；只是物理表结构换成“新表 + 旧表待迁移”。

### 迁移预算

run_migrate_budget() 每次只迁移一个旧 facility：

```cpp
  void run_migrate_budget() {
    if (!old_)
      return;
    if (migrate_progress_ >= old_->facilities.size()) {
      old_.reset();
      migrate_progress_ = 0;
      global_metrics().on_resize_finish();
      return;
    }
```

迁移某个旧 facility 时，它不依赖旧 Router，而是直接扫描旧 facility 的 tail 和 tiers 中所有 slot，通过 recover_key_from_slot() 恢复 key：

```cpp
// 搬迁一个 facility
Facility &of = old_->facilities[migrate_progress_];
// 收集 keys（不依赖 old 的 router，以防一致性问题）
const size_t oidx = migrate_progress_;
std::vector<uint64_t> keys;
if (of.tail_owned) {
  for (size_t si = 0; si < of.tail_owned->slots.size(); ++si) {
    if (!of.tail_owned->slots[si].has_value())
      continue;
    if (auto rk = recover_key_from_slot(*of.tail_owned, si, oidx, *old_);
        rk)
      keys.push_back(*rk);
```

恢复出来的 key 再按新 active_.N / active_.K 重新路由，并用当前的 kkick_insert() 插入新表：

```cpp
for (uint64_t k : keys) {
  // 迁移期间：如果 key 已在 active（可能被新插入覆盖），跳过
  uint64_t gx = pi_.pi(k);
  Facility &nf = facility_for_key(active_, gx);
  const size_t nb = route_bucket_for(active_, gx);
  if (router_at(nf, nb).locate(k).first.has_value())
    continue;
  ensure_tail(active_, nf);
  (void)kkick_insert(nf, nf.tail, k,
                     facility_index_of(active_, &nf));
```

迁移完成一个 facility 后，会清空旧 facility，避免后续查询/删除再次看到它：

```cpp
// 清空旧 facility，标记完成（保留 D 桶数，避免迁移窗口内 query 仍路由到已处理
// facility 时 vector 为空导致越界）
for (auto &rt : of.D)
  rt.clear();
of.tiers.clear();
of.tail_owned.reset();
of.tail = nullptr;
migrate_progress_++;
```

### 迁移期间的读写语义

迁移窗口内同时存在 active_ 和 old_。

查询：先查新 active_，没找到再查 old_。

```cpp
if (!r.found && old_) {
  const Facility &of = facility_for_key(*old_, gx);
  const size_t ob = route_bucket_for(*old_, gx);
  auto [oloc, osteps] = router_at(of, ob).locate(key);
  r.router_probe_steps = std::max(r.router_probe_steps, osteps);
  r.found = oloc.has_value() &&
            slot_pi_matches(*oloc->first, oloc->second,
```

插入：先查 active_，再查 old_，任一存在都认为已存在；新 key 只插入 active_。

删除：先删 active_，再删 old_。这样迁移期间无论 key 还在旧表，还是已经迁到新表，都能删到。

实际扩缩容逻辑全部写在 src/ht.cpp 的 maybe_resize_locked()、start_migration()、run_migrate_budget() 里。

总结：当前实现的扩容/缩容是渐进式 rehash。触发后新表立即接管写入，旧表保留用于查询和删除；每次插入/删除迁移一个旧 facility，直到 old_ 全部清空。优点是避免一次性全表搬迁；限制是如果之后没有插入/删除，迁移不会继续推进，查询也不会消耗迁移预算。

## 参数设计

当前系统参数分两类：外部输入参数和内部派生参数。

外部主要是 TableParams：

```cpp
struct TableParams {
  uint64_t n = 10000;        // 规模参数（N 会取 2^p，使 N <= n <= 2N）
  int k = 2;                 // 保留（对接设计文档的 k-kick 层数参数）
  double load_factor = 0.90; // 保留

  // 用于可逆置换 pi 以及其他哈希/随机选择的种子
  uint64_t seed1 = 0;
  uint64_t seed2 = 0;
  uint64_t seed3 = 0;
```

含义是：

- n：规模提示，不是当前元素数。初始化时用它推导表规模 N。代码实际是 N = next_pow2(max(2, n))，也就是取不小于 n 的 2 的幂。
- k：k-kick 参数。当前实现里主要影响两处：kkick_insert() 最多踢出/探测多少轮，以及 Facility::max_tier 的层级上界。
- load_factor：扩容阈值。n_ > N * load_factor 时扩容，默认 0.90。
- seed1/2/3：生成 Feistel 可逆置换 pi() 的种子，影响 key 到 π(key) 的分布。
- storage/mysql_*：服务端持久化配置，和哈希结构本身关系不大。

HTTP 初始化接口还额外限制：

- n 范围：10 .. 5,000,000
- k 范围：0 .. 8
- load_factor 必须在 (0, 1)，但前端初始化当前主要用默认值。

内部派生参数：

- N：表规模，2 的幂。
- K：每个 facility 的 Router 数量，也是局部桶数。由 choose_K(N) 得到，大致取 ceil(log2(N)^2)，限制在 [64, 4096]，再向下取 2 的幂，并且不超过 N。
- facilities = N / K：facility 个数。
- cubby_capacity(K, N, tier)：每个 cubby 容量，约等于 K / (iterated_log(N, tier+1)^2)，再限制为最小 4、最大 K。
- max_tier = k：facility 内 tier 层数上界。

所以整体结构大致是：

text
n -> N
N -> K
N/K -> facilities
K,N,tier -> cubby_capacity
k -> max_tier + kkick_insert 的移动上界

### 功能测试数据

### 性能测试数据

perf/bench.cpp 是性能基准。默认参数：

- n_insert = 50000
- n_rand_queries = 200000
- table_n_hint = max(10000, n_insert * 4)
- k = 4
- load_factor = 0.90
- OTSH_SEED 控制随机 key 序列，默认 1。

它测 5 个阶段：

- insert：插入 n_insert 个唯一随机 key。
- query_seq：顺序查询所有已插入 key，全是命中。
- query_rand：随机抽已插入 key 查询，全是命中。
- delete：随机删除一半 key。
- query_mixed：混合查询，约 70% 查询剩余 key，约 30% 查询随机 miss。

每个阶段的数据含义：

- ops：该阶段操作数。
- elapsed_ms：该阶段从开始到结束的经过时间（墙钟，steady_clock），毫秒。
- qps：吞吐，ops / elapsed_time。
- lat_ns_avg：单次操作平均延迟。
- p50/p95/p99：单次操作延迟分位数。
- max_ns：最大单次延迟，主要看尾延迟和偶发 resize/rebuild 影响。

### Metrics 指标含义

性能输出和 /api/stats 里会看到这些累计指标：

- state.n：当前元素数量。
- state.N：当前 active 表规模。
- state.K：每个 facility 的 router bucket 数。
- state.facilities：active facility 数量。
- insert_moved_total/max：插入过程中的 kick/搬移次数总和和最大值。
- delete_moved_total/max：删除时从 tail 回填等搬移次数。
- router_steps_total/max：Router 查询路径步数累计和最大值。
- meta_bits_total/max：记录到 metrics 的元数据 bit 估算，来自 Router bits 和 MiniArray bits。
- rebuild_down/up：后台 tier 重建事件次数。
- resize_start/finish：扩缩容开始/完成次数。
- ht_insert/query/delete 的 p50_ns/p99_ns/max_ns：基于 metrics 桶估算的延迟分布。

需要注意：meta_bits_total 是“每次操作记录的空间指标累计”，不是当前真实内存占用；space_bits 里的估算也比较粗，尤其 /api/stats 里 payload_bits 目前按 64 * n 估算，不能精确反映商压缩后的真实 bit 数。它更适合看趋势，不适合作为严格空间证明。
