\chapter{引言}

\section{研究背景及意义}

哈希表通过哈希函数将关键字映射到数组位置，是数据库索引、键值存储、编译器符号表和运行时系统中常用的基础结构。开放寻址哈希表具有连续存储和查询路径短等优点，但在高负载下容易受到冲突影响。工程实现通常通过预留空槽降低冲突概率，这会带来额外空间消耗。因此，哈希表设计中的一个核心问题是：在保持操作时间可控的前提下，如何减少单个元素对应的附加空间。

论文 \emph{On the Optimal Time/Space Tradeoff for Hash Tables} 对这一问题给出了理论化刻画\cite{pagh2021otsh}。该文指出，在常数时间查询的约束下，若允许插入和删除操作承担 \(O(k)\) 的元素交换成本，则每个元素的额外空间可由 \(O(\log\log n)\) 比特降低到 \(O(\log^{(k)}n)\) 比特\footnote{本文中讨论的空间开销均以比特为单位。}\footnote{记号 \(\log^{(k)}n\) 表示对 \(n\) 连续取 \(k\) 次对数：\(\log\log\cdots\log n\)（共 \(k\) 个 \(\log\)）。}。该结论说明，哈希表并非只能在“低负载高空间”和“高负载高冲突”之间取舍，还可以通过受控迁移、紧凑路由和商压缩等机制重新分配时间与空间成本。本文在该理论工作的基础上实现一个 C++ 原型，目的在于把论文中的主要结构落到可运行系统中，并通过测试和微基准观察其核心路径的行为。

\section{国内外研究现状}

哈希表的经典实现主要包括链地址法和开放寻址法。链地址法通过链表或桶结构处理冲突，实现简单但指针和分配开销较明显；开放寻址法将元素直接放入数组槽位，缓存局部性较好，但性能对负载因子和探针序列较敏感\cite{cormen2009introduction}。在此基础上，布谷鸟哈希通过多个候选位置和踢出过程提高查询确定性\cite{pagh2004cuckoo}，完美哈希则面向静态集合给出更强的查找保证\cite{fredman1984sparse}。这些方法各自侧重查询效率、构造代价或空间利用率，但动态场景下的空间下界仍然是较难同时兼顾的问题。

近年来，相关研究更加关注哈希表的位级空间开销。与只比较装载率不同，位级分析要求明确记录每个元素为支持查询、更新和恢复所额外保存的信息量。本文参考的最优哈希表理论正是在这一背景下提出：它将元素放置抽象为球槽分配问题，并用 k-kick 树、Local Query Router、MiniArray 和商压缩组合出新的时间—空间权衡。本文不试图重新证明该理论结果，而是围绕其中可工程化的结构进行实现与验证。

\section{研究路线}

本文尝试落地论文 \emph{On the Optimal Time/Space Tradeoff for Hash Tables} 中的主要结构。系统使用 \texttt{Facility--Cubby--Slot} 组织局部存储；每个 Facility 维护一组 Router，用于把键定位到具体 Cubby 和槽位；槽位中保存商压缩后的 payload，MiniArray 保存恢复键所需的变长元数据；扩缩容采用 active/old 双表渐进迁移。实现侧重点是打通插入、查询、删除、键恢复和迁移路径，并记录踢出次数、路由步数、元数据位数等指标。

本文围绕三个问题展开：其一，原型系统能否在批量插入、命中查询、删除、重复插入和迁移窗口中保持一致语义；其二，实际运行中插入移动次数、Router 访问步数和逻辑元数据位数呈现何种特征；其三，当前 C++ 实现相对于论文结构做了哪些简化，这些简化对实验结论的解释有何限制。第 \ref{ch:theo_base} 章说明相关理论基础，第 \ref{ch:system} 章给出系统实现，第 \ref{ch:exp} 章报告测试与微基准结果，第 \ref{ch:concl} 章总结本文完成的工作和后续可改进方向。
