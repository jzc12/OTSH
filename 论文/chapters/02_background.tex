\section{开放寻址中的两个代价}
\label{sec:balls-bins}
开放寻址哈希表把元素直接放在数组槽中。对一个键 \(x\)，哈希函数给出探针序列
\[
  h_1(x), h_2(x), h_3(x), \ldots
\]
查询时沿序列寻找目标槽位，插入时通常选择某个空槽。若负载较低，这一路径很短；若负载接近满表，冲突会让路径变长。《On the Optimal 
Time/Space Tradeoff for Hash Tables》把这个过程抽象成“球与槽”的在线分配：槽位数为 \(n\)，任意时刻最多有 \(n\) 个球，每个球
只能被放在自己探针序列中的某一项，更新操作允许移动已经放入的球。

在该模型中，最重要的不是单个哈希函数如何计算，而是两个代价的关系。
\begin{itemize}
  \item \textbf{探针复杂度。} 若球 \(x\) 最终放在探针序列的第 \(i\) 个位置，其复杂度记为 \(\Theta(1+\log i)\)。直观上，查询若
  要直接跳到这个位置，系统需要知道“它用了第几个候选槽”，记录该索引需要约 \(\log i\) 位。因此，平均探针复杂度可以看作额外元数据
  空间的代理量。
  \item \textbf{交换成本。} 一次插入或删除可能把已有元素踢到新位置。被移动元素的数量反映了更新操作的代价，也会影响实际系统中延迟。
\end{itemize}

传统开放寻址常把冲突处理放在探针序列本身，例如线性探测、二次探测或双重哈希\cite{knuth1998taocp3}。论文的思路更主动：允许更新时重新安排少量已有元素，用有限
交换换取更短的平均探针复杂度。本文后续实现中的 \texttt{kick\_count}、移动次数和路由步数，正是对这两个理论量的具体观测。

\section{k-kick 树的分层思想}
k-kick 树不是必须按指针实际存储的一棵树，而是一套对数组槽位进行分层划分和优先级驱逐的规则，逻辑上的管理方式。设槽位总数为 \(n\)，逻辑
上把数组划为 \(k+1\) 层：第 \(0\) 层是整个数组，第 \(1\) 层划为大小约 \((\log n)^6\) 的 bin，第 \(2\) 层继续缩小到 \((\log\log n)^6\) 量
级，依此类推。第 \(i\) 层 bin 的大小记为
\[
  s_i = \Theta\bigl((\log^{(i)} n)^6\bigr).
\]
越深的 bin 越小，也越接近一个元素的“理想局部位置”，优先级越高。

每个球 \(x\) 在插入前由随机性确定三类信息。第一，它选择一个叶层 cubby，并由此确定所有祖先 cubby \(g_0(x),g_1(x),\ldots,g_k(x)\)。第二，
它得到一个最高允许深度 \(s(x)\)，该值决定优先级，深层元素优先留在更小的局部区域。第三，它的探针序列按从深到浅的顺序遍历这些 cubby：先
看 \(g_k(x)\)，再看 \(g_{k-1}(x)\)，最后到 \(g_0(x)\)。

\section{级联驱逐与 \(O(k)\) 交换}
插入过程可以描述为“寻找一个未饱和 cubby，然后把更低优先级元素向上层推”。对新球 \(x\)，算法从较深层开始检查其祖先 cubby。若某个 cubby 有
空槽，则直接放入。若 cubby 已满但未饱和，说明其中至少有一个不应该占据该层优先位置的低优先级球；算法将其踢出，让 \(x\) 占据该槽。被踢出
的球再到自己允许的较浅层继续寻找位置，必要时继续踢出其他低优先级球。这一过程最多形成 \(k+1\) 级级联。

这一过程的关键在于层数受 \(k\) 控制。每一次踢出都会把问题推向更浅层，同一层不会无限反复，因此一次插入最多形成 \(k+1\) 级级联。删除操
作也可以通过从 tail 或其他局部结构迁移元素补洞，保持类似的受控移动。理论分析给出的结果是：在适当随机假设和不变量维护下，更新成本
为 \(O(k)\)，平均探针复杂度为 \(O(\log^{(k)} n)\) 量级\cite{pagh2021otsh}。

对本文实现来说，这个结论有两层意义。第一，系统需要记录每次插入和删除到底移动了多少元素，用来判断工程路径是否出现异常长链。第二，系统要把
“深层局部优先”的思想落实到 Cubby、tier 和 tail 的组织中，而不是仅依赖普通开放寻址的首次空槽策略。

\section{从理论量到系统指标}
理论分析中的探针复杂度和交换成本不能直接等同于某个 C++ 计时结果。实际延迟还受到缓存、锁、HTTP 编解码、MySQL 写入和操作系统调度影响。因此
本文在实验章采用“可观测代理量”的方式验证原理实现。

\begin{itemize}
  \item 路由步数对应查询从键到 \((\mathrm{cubby},\mathrm{slot})\) 的查找成本；
  \item \texttt{kick\_count} 与移动次数对应插入、删除中的交换成本；
  \item \texttt{meta\_bits} 与 Router 编码长度对应额外元数据空间；
  \item tier 计数、cubby 容量和 tail 状态对应分布不变量；
  \item 延迟分位数用于观察工程实现是否出现不可接受的长尾。
\end{itemize}

\section{本章小结}
本章从开放寻址的动态冲突问题出发，说明了探针复杂度与交换成本的含义，并介绍了 k-kick 树通过分层 cubby 和级联驱逐获得 \(O(k)\) 更新代
价的基本机制。下一章将在此基础上讨论完整系统设计：如何用 MiniArray、Local Query Router、Facility、商压缩和后台重建把这些理论部件连接
为可运行的哈希表。
