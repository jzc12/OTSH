\chapter{问题模型与理论基础}
\label{ch:theo_base}

\section{探针复杂度与球槽分配模型}
开放寻址哈希表可以抽象为在线球槽分配问题。设表中有固定数量的槽位，每个槽至多容纳一个元素；元素支持动态插入和删除，任意时刻的元素数不超过槽位数。对每个键 \(x\)，哈希过程为其生成一条固定随机探针序列 \(h_1(x),h_2(x),\ldots\)，元素只能被放置在该序列给出的候选槽位中。

若元素最终存放在其探针序列的第 \(i\) 个候选位置，则探针复杂度定义为 \(\Theta(1+\log i)\)。该量可理解为查询时记录“使用第几个候选位置”所需的比特数，因此直接对应元数据的空间开销。该模型下有两个需要同时控制的量：平均探针复杂度决定整体附加空间，交换成本表示一次更新中被搬移的元素数量。最优哈希表理论的关键问题，就是在允许有限交换的条件下降低平均探针复杂度。

\section{k-kick 树分层驱逐思想}
k-kick 树不是必须以指针树形式存在的数据结构，而是一种分层放置与受控驱逐规则。它将槽位划分为 \(k+1\) 个逻辑层级，不同层级的 Cubby 尺寸随深度递减。深层 Cubby 更靠近元素的随机偏好位置，因此具有更高放置优先级；浅层 Cubby 范围更大，用于接收在深层无法放置的元素。

插入前，哈希过程为元素确定一条从深层到浅层的 Cubby 谱系，并确定其允许使用的最大深度。插入时优先尝试深层 Cubby；若目标 Cubby 已满但尚未饱和，则可以驱逐其中优先级较低的元素，将当前元素放入该位置，被驱逐元素再向更浅层寻找合法位置。由于每次驱逐都沿层级向上推进，级联过程受到层数限制。

层数由参数 \(k\) 控制，因此单次更新中的迁移次数可限制在 \(O(k)\) 量级。理论结果表明，在这一交换成本下，平均探针复杂度可降至 \(O(\log^{(k)}n)\)。这一结论是后续系统实现中记录 \texttt{kick\_count}、区分理论 k-kick 与工程版踢出链的依据。

\section{紧凑元数据存储核心思想}
分层放置降低了探针复杂度，但查询仍需要知道元素最终所在的 Cubby 和槽位。理论结构为此引入 MiniArray 与 Local Query Router 两类组件，分别处理变长元数据存储和局部键到槽位的映射。

MiniArray 用于保存 Cubby 内不同长度的元数据条目。理论方案利用内存 B 树和查表方法，将变长比特串打包存储，并支持均摊常数时间读写。其空间开销与条目总比特数线性相关，避免为所有槽位按最大长度预留空间。

Local Query Router 负责小规模键集合的本地定位。它将键映射为随机比特串，用二进制前缀树区分不同键，再把树结构和叶子上的槽位信息编码为连续比特流。由于每个 Router 只管理局部少量元素，前缀树的期望规模较小，可在低空间开销下完成键到 Cubby、Slot 的映射。本文第三章中的 Router 和 MiniArray 均参照这一接口语义实现，但更新策略采用了工程化简化。

\section{分层架构整体设计思想}
本文实现参照 Facility--Cubby--Slot 三级架构组织数据。Facility 负责表的局部分区与路由，Cubby 负责局部存储和元数据维护，Slot 对应实际数组位置。

Cubby 是局部存储单元，内部包含存储槽、变长元数据和空闲槽检索结构，承担元素放置、局部驱逐和键恢复所需信息维护。Facility 聚合多个 Cubby，并通过局部 Router 将键定位到目标 Cubby 与槽位。理论结构还引入多 tier Cubby 和 tail 机制：大部分非 tail Cubby 尽量保持满载，插入和删除造成的波动集中由 tail 承担。

为适配元素数量变化，理论结构用分布不变量约束各 tier Cubby 的数量，并通过后台拆分、合并平摊重建代价。本文实现保留 Facility、tier、tail、Router 和双表迁移等主要接口，但未完整实现论文中的全部后台重建规则。因此，第二章描述的是理论依据，第三章将进一步说明当前代码中的实现边界。

\section{商压缩的信息论最优思想}
若哈希表在槽位中完整保存原始键，则其中一部分信息与寻址过程重复。商压缩利用随机置换后的低位参与路由这一事实，只在槽位中保存无法由位置和元数据恢复的剩余位。

具体而言，\(\pi(x)\) 的低 \(\log N\) 位用于确定 Facility、Router 桶和局部偏好位置；槽位只保存去掉这部分后的 payload。为了恢复完整 \(\pi(x)\)，元数据记录偏好槽位与实际槽位之间的差值，以及局部桶内的中间位。结合 Facility 编号、槽位下标和 payload，即可重建置换值，再通过 \(\pi^{-1}\) 得到原键。第三章将按照当前代码中的 \texttt{payload}、\texttt{mid\_bits} 和 \texttt{delta} 说明该过程。

\section{算法层次全景总结}
上述理论结构可归纳为三层：
\begin{itemize}
  \item Cubby 层负责元素放置、空闲槽检索、局部驱逐和变长元数据编码；
  \item Router 层负责把键映射到目标 Cubby 与 Slot，并为查询提供局部定位信息；
  \item Facility 层负责组织 Cubby、维护 tail、承载扩缩容和重建过程中的局部管理。
\end{itemize}
本文后续实现以这三层为结构参照，同时明确区分理论结构和工程原型之间的差异。
