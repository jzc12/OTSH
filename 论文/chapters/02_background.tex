\chapter{相关工作与背景}

\section{哈希冲突处理与探测代价}
哈希表的查询与插入性能与冲突处理策略紧密相关。链式哈希通过链表或其他容器存储同一桶内元素，插入较为直接，但在高负载或热点分布下可能产生较长链并引入额外的指针追踪开销。开放定址哈希将元素存放在表内槽位，通过线性探测、二次探测或双重哈希等策略寻找空槽；其优点是空间局部性较好，但在高负载时探测次数会显著上升，并出现聚集效应。

一些更强的结构通过置换或多选择策略控制查询代价。例如 cuckoo hashing 通过为每个键提供多个候选位置，并在插入时通过“踢出链”置换已有元素，通常能提供较好的查询上界；Robin Hood hashing 则通过均衡探测距离改善尾延迟表现。上述方法在工程中广泛应用，但其查询代价上界、空间开销和实现复杂度之间存在取舍。

\section{时间/空间折中与 OTSH 思想}
理论工作《On the Optimal Time/Space Tradeoff for Hash Tables》\cite{pagh2021otsh}讨论了在空间开销接近线性、负载较高的条件下，如何构造支持常数次探测查询的数据结构。其核心思路之一是改变“探测”的粒度：不再将一次探测限定为单个槽位访问，而是允许一次探测对应一个连续下标区间的扫描（在工程上可对应一次范围查询或一次 cache-friendly 扫描）。在这种意义下，如果每次查询只需访问常数个连续区间，即可获得接近常数的查询代价。

本文实现采用的具体结构将槽位划分为 bin、mini-bin 与 fallback 区域，使得每个键的候选位置集中在两个连续区间内，并通过有界深度的置换策略处理冲突。

\section{指纹过滤}
为降低验证成本，工程实现中常采用指纹（fingerprint）作为轻量过滤：为每个键计算较短的哈希摘要，查询时先比较指纹，再在必要时比较完整键。本文实现为每个槽位存储 16 位指纹，且约定指纹为 0 表示空槽（当计算结果为 0 时将其映射为 1），以便快速判断占用情况并减少无效比较。

\section{数据库化实现的特点}
与纯内存实现不同，本文将槽位落在 MySQL 表中，利用事务与行锁实现并发安全，并将“区间探测”对应为 SQL 的范围查询。数据库化实现的优点是便于持久化、便于在受限内存环境中进行大规模实验与统计；同时也带来额外的往返开销与事务开销，需要在实现中尽量减少 SQL round-trip（例如在插入中对单个 bin 进行一次锁定并在内存中模拟踢出链，再批量写回）。
