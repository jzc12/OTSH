\chapter{系统验证与实验设计}
\label{ch:exp}

\section{实验目标与评价口径}
本章验证第三章实现的核心路径，包括功能正确性、操作延迟、累计指标和结构快照。由于当前系统是理论结构的工程原型，实验不用于替代第二章中的概率分析，也不用于证明空间最优性；实验目标限定为三个问题：插入、查询、删除和重复插入是否满足预期语义；Router、MiniArray、商压缩和工程版 k-kick 对延迟表现有何影响；运行过程中记录的移动次数、路由步数和元数据位数能否反映实现中的主要开销。

\section{功能正确性测试}
\texttt{otsh\_tests} 是系统的自动化回归入口。该程序直接调用 \texttt{HashTable}，不经过 HTTP 层和 JSON 编解码流程，测试对象集中在哈希表核心结构。测试流程包括：
\begin{itemize}
  \item 初始化哈希表，并检查返回状态与表状态中的 \(N\)、Facility 数量是否有效；
  \item 使用固定随机种子生成唯一键集合，执行批量插入，要求每个新键返回 \texttt{inserted=true}；
  \item 对全部已插入键执行查询，要求返回 \texttt{found=true}；
  \item 随机删除一半键，要求删除操作成功；
  \item 再次查询已删除键，要求返回未命中；
  \item 对剩余键重复插入，要求返回 \texttt{inserted=false}，验证幂等语义。
\end{itemize}

测试结束时输出单行 JSON，字段包括 \texttt{all\_passed}、通过和失败数量、各测试用例名称与结果，以及 \texttt{ops\_insert}、\texttt{ops\_query}、\texttt{ops\_delete}、\texttt{insert\_moved\_total}、\texttt{delete\_moved\_total} 等摘要指标。结果写入本地日志。

\section{性能基准}
\texttt{otsh\_perf} 用于测量内存态哈希表路径。该程序不链接 MySQL，也不经过 HTTP JSON 编解码，避免存储层和接口层影响插入、查询、删除本身的耗时。程序支持命令行参数 \texttt{n\_insert}、\texttt{n\_rand\_queries}、\texttt{table\_n\_hint}，并通过环境变量 \texttt{OTSH\_SEED} 固定随机种子。

每个阶段记录操作数、吞吐、平均延迟与 p50 分位数。结果输出到 \texttt{stderr}，单行 JSON 输出到 \texttt{stdout}，便于脚本收集到 \texttt{perf/results/}。本文实验参数为 \texttt{n\_insert}${}=5\times10^4$、\texttt{n\_rand\_queries}${}=2\times10^5$、\texttt{table\_n\_hint}${}=2\times10^5$、\texttt{OTSH\_SEED}${}=1$。

\section{指标与数据库观测}
操作级指标以 \texttt{SqlMetricRow} 形式产生，核心字段包括 \texttt{operation\_type}、\texttt{probe\_count}、\texttt{kick\_count}、\texttt{latency\_ns}、\texttt{cubby\_tier} 和 \texttt{snapshot\_id}。存储层先把指标放入内存缓冲，再批量写入 \texttt{otsh\_op\_metrics}，避免高频操作逐条落库造成过大扰动。

这些字段与第 \ref{ch:theo_base} 章中的探针复杂度和交换成本相对应。\texttt{probe\_count} 记录局部 Router 的访问步数，\texttt{kick\_count} 记录更新过程中的移动次数，\texttt{latency\_ns} 记录工程延迟，\texttt{cubby\_tier} 用于区分不同层级 Cubby 上的操作行为。后续分析可以按操作类型统计延迟分位数，也可以按 tier 或快照比较路由步数和移动次数。

结构快照由 \texttt{HashTable::visit\_structure} 驱动。系统将 Facility、Cubby、Slot 和 tier 聚合信息写入相应数据表：\texttt{facility} 记录 tail 所在 Cubby，\texttt{cubby} 记录 tier、capacity、size 和 \texttt{is\_tail}，\texttt{slot\_snapshot} 记录槽位是否占用及 \(\pi(x)\) 形式的 key hash，\texttt{tier\_stat} 记录每个 Facility 在各 tier 上的 Cubby 数量。这些表可用于检查 tail 是否唯一、非 tail Cubby 是否接近满载、tier 数量是否偏离目标分布。

指标写入属于观测路径，不属于哈希表核心算法。性能基准程序绕开 MySQL 和 HTTP 层，是为了减少观测系统对热路径延迟的干扰；数据库指标更适合用于实验后分析、结构审计和可视化展示。
