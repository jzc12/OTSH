\chapter{总结与展望}
\label{ch:concl}

\section{工作总结}
本文围绕最优时间/空间折中哈希表的工程实现展开。理论部分从开放寻址的冲突问题出发，使用“球与槽”模型解释探针复杂度和交换成本，并介绍 k-kick 树如何通过分层 cubby、随机优先级和级联驱逐把更新代价控制在 \(O(k)\) 量级。系统设计部分进一步说明，单靠一个探针函数无法达到目标，必须把局部驱逐、紧凑元数据、局部路由、商压缩和后台重建组合起来。

系统以 \texttt{HashTable} 为核心接口，内部按 Facility、Cubby 和 Slot 分层组织；使用可逆置换 \(\pi\) 支持商压缩和键恢复；使用 \texttt{Router} 管理键到槽位的局部映射；使用 \texttt{MiniArray} 保存变长元数据；使用 \texttt{FreeSlotTree} 管理空槽；使用 \texttt{RebuildScheduler} 和双表迁移处理 tier 重建与扩缩容。服务层通过 HTTP 暴露初始化、插入、查询、删除、统计、结构导出和参数实验接口，存储层通过 MySQL 记录键、指标和结构快照。

验证部分没有把原型性能简单等同于论文理论上界，而是建立了更适合当前系统的证据链。 \texttt{otsh\_tests} 检查核心语义，\texttt{otsh\_perf} 测量无数据库热路径，HTTP 接口验证服务化流程，\texttt{Metrics} 与 \texttt{otsh\_op\_metrics} 记录路由步数、踢出次数、移动次数、延迟和重建事件，结构快照则用于观察 Facility、Cubby、Slot 和 tier 分布。

\section{不足之处}
当前系统与论文中的完整理论结构存在差距。
\begin{itemize}
  \item k-kick 树没有完全实现论文级分层饱和判定和严格优先级驱逐，而是在插入路径中以有限踢出、tail 和局部搬迁近似；
  \item MiniArray 采用连续 bit buffer 与重打包实现，尚未实现 packed B-树、查找表加速和严格均摊常数更新；
  \item Router 使用二叉 trie 与重建策略表达局部路由语义，尚未实现理论中的紧凑编码常数时间更新；
  \item 实验章强调验证方法和可采集指标，若要形成最终定量结论，还需要在固定硬件、固定编译参数和多组随机种子下补充完整运行数据。
\end{itemize}

\section{后续工作}
后续优化可以沿三条线推进。第一条是算法结构补全：把 k-kick 树的层级饱和、优先级和级联驱逐从当前插入路径中抽出，形成独立模块，并与 tier 重建共享不变量。第二条是紧凑结构优化：用 packed B-树或分块 bitset 改写 MiniArray，用增量维护的压缩 Trie 改写 Router，减少更新后的整体重建。第三条是实验体系完善：补齐不同 \(n\)、\(k\)、负载因子和随机种子下的批量实验，把延迟分位数、元数据位数、移动次数和 tier 分布整理成稳定图表。

此外，当前结构快照已经能导出 Facility、Cubby 和 Slot 信息，后续可以接入离线分析脚本或可视化页面，把一次实验中的操作指标和结构状态放在同一时间线上观察。这会有助于区分算法本身的代价、HTTP/数据库 I/O 的代价以及扩缩容迁移造成的短时波动。

\section{结束语}
本文完成的工作可以概括为：把一套以空间最优为目标的哈希表理论结构，拆解为可实现的系统模块，并为每个关键原理保留验证入口。虽然当前原型还没有达到论文中所有微观结构的最优复杂度，但它已经能够展示论文思想在工程系统中的主要形态，也为后续继续完善算法细节和实验数据打下基础。
