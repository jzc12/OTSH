## 一、项目概述

### 1.1 目标

复现论文 On the Optimal Time/Space Tradeoff for Hash Tables 中提出的哈希表结构，在工程上实现其核心性质与操作接口。

- 查询：在“区间扫描次数”意义下，单表内每次查询最多查两个连续下标区间（偏好 mini-bin 与同一 bin 的 fallback），与“常数次探测”一致；若将 mini-bin 内逐槽比较计入代价，则与结构参数相关。
- 插入：有界深度 k-kick，深度超过 k 时尝试将键仅放入 fallback 等兜底路径（见第四节）。
- 空间：由目标规模 n、load_factor 与派生的 bin 尺寸共同决定；每键在库中存完整键值与 16 位指纹（见第九节）。
- 功能：插入、查询、删除、在线扩容（双表与元数据）、HTTP 与前端实验接口。

### 1.2 核心思想

结构可概括为：全局分桶（bin）与桶内分 mini-bin，加固定大小 fallback 区域，再用有界深度置换（k-kick）解决冲突。

- 全局划分为若干 bin；每个 bin 内划分为多个 mini-bin，尾部为 fallback 槽（本实现中每 bin 的 fallback 大小为常数 8）。
- 每个 key 经哈希属于唯一 bin，再经独立哈希得到偏好 mini-bin；插入与查询均围绕该 bin 内局部完成。
- k-kick 将冲突元素在有限步内重排；本实现中踢出槽为确定性函数（依赖指纹与深度），避免数据库端高代价随机。
- 查询在逻辑上只检查“偏好 mini-bin 对应下标范围”和“该 bin 的 fallback 下标范围”。

### 1.3 本项目的实现方式

| 项目 | 说明 |
| --- | --- |
| 存储与计算 | MySQL 或 MariaDB，槽位与元数据、事务、行锁均落库 |
| 服务进程 | C++20，可执行文件 otsh_backend，HTTP 由 cpp-httplib 提供，JSON 由 nlohmann_json 解析 |
| 连接库 | 官方 MySQL/MariaDB 客户端，需 mysql.h 与对应库文件 |
| 前端 | Vue 3、TypeScript、Vite，通过 /api 调用后端（开发时常用代理） |

与纯内存论文原型的区别：不驻留整表大数组，逻辑上的连续下标与表主键 idx 一一对应；k-kick 的“选槽”在实现上为确定性，便于在 SQL 上可重复执行与调试。

## 二、整体结构设计

### 2.1 全局结构

```text
HashTable（逻辑）
├─ 元信息（n, k, load_factor, 三枚种子, total_bins, 扩展开关与进度等）
└─ 共 B 个 Bin
   ├─ 若干 MiniBin 槽位（主存，连续下标）
   └─ Fallback 区（每 bin 固定槽数，本实现为 8）
```

### 2.2 参数定义（理论目标与本实现对应关系）

| 参数 | 理论 | 本实现中的说明 |
| --- | --- | --- |
| n | 数据规模 10^4 量级起 | 初始化时由用户指定；接口校验约 10 到 5×10^6 |
| k | 1 或 2、3、4 等，控制重排深度上界 | 由用户指定；接口允许 0 到 8，k=0 即不允许加深递归 |
| mini_bin_size, num_mini_bins | 同阶，约 O(log^(k) n) | 由函数 ilog_iter(n, max(1,k)) 派生，再限制在 4 到 64 之间，避免过小无结构或过大查询区间 |
| fallback_size | 常数 | 固定 8 |
| bin_size | mini 区加 fallback 区 | mini_bin_size × num_mini_bins + fallback_size |
| total_bins | 与容量和 bin_size 相关 | 由目标容量与 bin_size 上取整得到，至少 1，在线扩容时可翻倍并持久化 |
| 指纹 | 16 位 | fp=0 表示空槽，非空键的指纹经哈希后恒不为 0，与建表列类型一致 |
| load_factor | 高负载 0.95 到 0.99 区间常见 | 用于由 n 推导目标槽位数，实际负载由已用行数与 capacity_slots 相除得到 |

由 n、k、load_factor 派生尺寸时，

本实现中核心关系为：先算单 bin 尺寸 bin_size，再算目标总槽位约 n/load_factor 的上取整，再算 total_bins 与 capacity_slots，细节与代码中 derive_sizes 一致。

### 2.3 映射关系

对任意 key 与三枚种子 seed1、seed2、seed3（与代码中 HashFunc 一致）：

```text
b = h_bin(key)   由 splitmix64(key 与 seed1 异或) 对 total_bins 取模
m = h_pref(key)  由 splitmix64(key 与 seed2 异或) 对 num_mini_bins 取模
```

在单个 bin 内，全局连续下标 idx 的构造方式为：

```text
bin 起点 offset_bin = b * bin_size
mini-bin 对应 idx 段：offset_bin + m*mini_bin_size 起，连续 mini_bin_size 个
fallback 对应 idx 段：接在全部 mini-bin 之后，连续 fallback_size 个
```

## 三、核心数据表示

### 3.0 逻辑与存储

```text
逻辑上：SLOT[0 .. total_bins*bin_size - 1] 一维数组

实现上：表 hash_table 每行一条槽位，主键 idx 即数组下标，不要求进程内开整块内存
```

### 3.1 槽位行（与当前实现一致）

本实现不单独使用 is_used 列；是否为空由 key 与 fp 共同表示：

| 列名 | 含义 |
| --- | --- |
| idx | 主键，全局下标 |
| key | 空表示槽空；有值时表示存储的 64 位无符号键，且在该列上有唯一约束 |
| fp | 0 表示空；非 0 为 16 位指纹 |

与早期仅含 fingerprint 的 Entry 草图相比，库中需保留完整 key 以支持按 key 定位与唯一性。

### 3.2 元数据

除理论参数 n、k、load_factor 与三枚种子外，在线扩容在表 ht_meta 中至少包含：当前 total_bins、is_resizing、new_total_bins、resize_progress 等，与代码中 load_or_init_meta 及 resize 流程一致。

### 3.3 哈希子程序（文字描述）

- h_bin、h_pref 均基于 splitmix64 的 64 位混洗，再对总桶数或 mini-bin 数取模。
- fingerprint 由第三枚种子经 splitmix64 取低 16 位，若为 0 则改为 1，以便 fp=0 专用于空槽。
- 在满 mini-bin 中选取踢出位置时，使用与指纹、深度、种子有关的确定性函数，再对 mini_bin_size 取模，与代码中 kick_slot 一致，而非在 SQL 中随机取行。

## 四、核心算法

### 4.1 插入与 k-kick

顶层插入：

```
InsertKeyIntoTable(key)：
1. 若本 key 已在表中，成功返回（幂等）
2. ok = InsertWithKick(key, 0)
3. 若 ok 为真，成功返回
4. 否则尝试仅把本 key 写入其所属 bin 的 fallback，成功则记为 placed_in_fallback_only 类结果
5. 若仍失败，则插入失败
```

核心递归过程：

```text
InsertWithKick(key, depth)：

1. 若 depth 大于 k，失败返回

2. 计算
   b = h_bin(key)
   m = h_pref(key)
   求出该偏好 mini-bin 在全局 idx 上的一段闭区间

3. 若该段内存在空槽：
   若本 key 曾放在 fallback 中，先清掉
   将 key 与指纹写入空槽
   成功返回

4. 若该段无空槽，在 mini 段内用确定性规则选出一个槽下标
   若该槽其实为空，写入 key 并成功返回
   否则被踢出元素记为 ev

5. 用当前 key 与指纹覆盖该槽
   若新 key 曾在 fallback，先清掉

6. 处理被踢出元素 ev
   6.1  将 ev 与 ev 的指纹写入 ev 所属 bin 的 fallback 中一个空位；若无空位，失败返回
   6.2  递归 ok = InsertWithKick(ev, depth+1)
   6.3  若 ok 为真，从 fallback 中去掉 ev 的临时占位
7. 返回 6.2 的结果
```

### 4.2 查询

```text
1. 计算 b、m，以及偏好 mini 段与当前 bin 的 fallback 段在 idx 上的范围
2. 在 mini 段做一次区间查询，探针计数加 1
   若存在行 key 与指纹与查询一致，返回找到
3. 否则在 fallback 段做一次区间查询，探针计数再加 1
   若存在行一致，返回找到
4. 未找到则返回未找到
```

### 4.3 删除

```text
在事务中可选执行一步扩容迁移
若处于扩容且 key 在新表中，则在新表清空该槽
否则在活动表中按 key 上锁，找到行则清为 key 空、fp=0
```

## 七、失败与重建

论文意义下低概率失败可通过换种重建整表；本项目中：

- 全表与元数据多由 /api/init 在可控条件下重建，用于实验冷启动，不等同于运行时自动重建的同一实现路径。
- 单次插入在极端下可能仅落入 fallback 或直接失败，错误信息或备注由返回 JSON 描述。

## 八、动态扩容

与内存中双指针表不同，实现采用两张物理表与元数据，大致如下。

### 8.1 触发

> 在插入路径上检查当前负载
> 若未处于扩容中，且 已用槽 / capacity_slots 不小于 0.99
> 在 ht_meta 中置 is_resizing、new_total_bins=2*原 total_bins、resize_progress=0

### 8.2 分步迁移与换表

```text
若 is_resizing 为真，确保存在 hash_table_next，行数约为 new_total_bins*bin_size，行结构与主表同
每次在写操作事务中可推进一个旧 bin：将该 bin 内已占用行读出 key，按新桶数重算 h_bin 后调用插入逻辑写入 next 表，并清空原表槽位
resize_progress 自增
当进度达到旧 total_bins 时，用 RENAME 等原子操作交换表名、删除中间名，并更新元数据
非扩容时插入写在活动表；扩容中写 next 表；查询可先 next 后旧表
```

### 8.3 与第四节的关系

读路径不强制推进迁移；

写路径在实现中在插入时触发 MaybeStartResize，

在扩容中每次插入会附带至少一步 ProcessResizeStep，与 ht.cpp、resize.cpp 中逻辑一致。

## 九、数据库部署

运行依赖本机或局域网的 MySQL/MariaDB；客户端已链接库。

### 9.1 设计目标

- 以数据库为存储与计算载体的主要形式，逻辑上保持连续下标与数组一一对应。
- 用事务、唯一约束与行锁支持并发下的正确性。

### 9.2 主表

与 init_table 中建表方式一致，示意如下：

```sql
CREATE TABLE hash_table (
  idx BIGINT PRIMARY KEY,
  key BIGINT UNSIGNED NULL,
  fp SMALLINT UNSIGNED NOT NULL,
  UNIQUE KEY uk_key (key)
);
```

空槽为 key 空且 fp=0。初始化时批量插入各行 idx 从 0 到 capacity_slots-1。

### 9.3 元数据

ht_meta 存 n、k、load_factor、种子、total_bins 及扩展开关与进度，见第三节与第八节。

