## 论文原理

### 最底层的核心问题：探针复杂度

任何增广开放寻址哈希表最终都可以归结为一个“球与槽”的在线分配问题。

#### 1.1 问题定义

想象你有 `n` 个槽（数组位置），每个槽最多放一个球（键）。球会动态地插入、删除，总数不超过 `n`。每个球 `x` 一出生就附带着一个固定的、随机的探针序列： 
`h₁(x), h₂(x), h₃(x), ...` 
它只能放在这个序列里的某个槽中。如果球 `x` 最终落在 `hᵢ(x)`，我们称它的探针复杂度为 `Θ(1 + log i)`。这个值直观上就是“为了在查询时直接跳到这个球的位置，最少需要额外记录的比特数”（因为你必须记录它用的是序列中的第几个位置，而存储索引 `i` 需要大约 `log i` 比特）。

球与槽方案要解决的就是在线地给每个球指派一个合法的槽位，并且允许在插入/删除时移动已有的球。两个性能指标：

- 平均探针复杂度：当前所有球的探针复杂度的平均值，对应最终哈希表的每个键的额外空间开销。
- 交换成本：一次插入/删除过程中移动的球的总数，对应操作的时间复杂度。

我们的目标就是在这两者之间找到最优的折中。

#### 1.2 上界：k-kick 树

> k-kick tree 是 “逻辑结构（analysis structure）”，不是物理结构

k-kick 树是一套分层优先级驱逐的方案，它用 `O(k)` 的交换成本换来了 `O(log^{(k)} n)` 的平均探针复杂度。

##### 分层结构
把 `n` 个槽组织成一棵深度为 `k+1` 的树，从第 0 层（全数组）到第 `k` 层，每一层把数组划分成大小呈指数级递减的“bin”：

- 第 0 层：整个数组，大小 `s₀ = n`。
- 第 1 层：bin 大小 `s₁ = Θ((log n)^6)`。
- 第 2 层：bin 大小 `s₂ = Θ((log log n)^6)`。
- ……
- 第 k 层：bin 大小 `sₖ = Θ((log^{(k)} n)^6)`。

这些 bin 之间有父子关系：每个第 `i` 层的 bin 包含在第 `i-1` 层的某个 bin 中。这就像一棵树，越往下叶子越小。

##### 球的随机偏好
每个球 `x` 在插入前就通过哈希函数随机确定了三件事：

1.  一个叶子 bin：随机选取第 `k` 层的一个 bin（称其深度 `k` 的 bin）。由此也就确定了它在每一层的祖先 bin。我们用 `g₀(x), g₁(x), …, g_k(x)` 表示球 `x` 在各层的 bin。
2.  一个最高允许深度：随机分配一个值 `s(x) ∈ [0, k]`，满足 `Pr[s(x) < i] = 1 / (log^{(i)} n)^2`。这个值决定了球的“优先级”：只允许被放在深度 `s(x)` 或更浅（即 bin 更大）的层。深度越深（bin 越小）优先级越高（因为越靠近叶子）。
3.  探针序列：它的探针序列按从深到浅的顺序遍历其所属的 bin：先遍列 `g_k(x)` 中的所有槽，然后是 `g_{k-1}(x)`，……，最后是 `g₀(x)`。这样，如果球最终在深度 `i` 的 bin 中，它的实际探针复杂度大约为 `Θ(log s_i) = Θ(log^{(i+1)} n)`。

##### 插入与级联驱逐
当插入一个新球 `x` 时，我们不直接选择探针序列中的第一个空位，而是采用贪心 + 优先权策略：

1.  确定 `x` 能使用的最高深度 `min(k, s(x))`。
2.  从深度 `k` 开始向下扫描，找到第一个未饱和的祖先 bin。一个 bin 饱和是指：它已满，并且里面所有的球深度都不浅于（即优先级不低于）这个 bin 的深度。
3.  假设找到的深度为 `i`，bin 为 `g_i(x)`。
    - 如果 bin 中有空位：直接放入。
    - 如果 bin 已满：因为 bin 未饱和，里面必定存在一个低优先级的球（深度 `< i`，即来自更上层 bin 的球）。我们随机选一个这样的球 `y`，把它“踢”出去，把 `x` 放到 `y` 的槽里。
4.  被踢出的 `y` 现在需要在它的所属深度（假设为 `i' < i`）的 bin `g_{i'}(y)` 中找一个新位置。这个过程与插入 `x` 类似：如果那个 bin 有空位则放入，否则再踢出一个更低优先级的球，形成级联驱逐。
5.  这个过程持续直到有一个球找到了空位（最终会因为深度 0 的 bin 是整个数组且必定有空位而终止）。

由于每次插入最多在每一层踢出一个球，而共有 `k` 层，所以交换成本 ≤ k+1。



### 中间层：紧凑元数据存储

现在我们知道如何把球放进槽里了，但光知道不行，还要高效地记录每个球在哪个槽，以及相关的控制信息。这就是第4章的作用。它提供了两个基本组件：MiniArray 和 Local Query Router。

#### 2.1 MiniArray：存储变长条目的紧凑动态数组

每个 cubby 里需要维护一个长度约为 polylog n 的数组，但每个元素（元数据）的比特长度不同，且会动态变化。如果简单地用指针或预先分配最大空间，浪费太大。MiniArray 就是用来解决这个问题的，它的总空间与所有条目的总比特数成线性关系，同时支持 O(1) 的读写。

实现方法：内存 B-树 + 查找表加速

- 把数组看作一个 B-树，每个节点的大小恰好是一个机器字（`Θ(log n)` 比特）。
- 叶子节点存储实际条目比特，内部节点存储指向孩子的偏移以及枢轴（孩子覆盖的索引范围）。
- 因为总条目数只有 polylog n，所以指针和枢轴都可以用 `Θ(log log n)` 比特，依然能打包进一个字里。树的高度是 O(1)。
- 所有在节点上的操作（读取枢轴、比较、提取孩子指针、分裂/合并节点）都可以通过预先构建的查找表在 O(1) 时间内完成。查找表以半个字的比特串作为输入，输出对应的处理结果，这一点利用了“四俄方法”思想。
- 当数组更新导致某个节点过长或过短时，像 B-树一样分裂或与兄弟合并，均摊 O(1) 时间。

#### 2.2 Local Query Router：为少量元素映射键到槽索引

在哈希表的顶层（Facility 的 D 数组）以及每个 cubby 的内部，我们都需要解决一个问题：给定一个键 `x`，立马知道它在哪个 cubby（或 cubby 内的哪个槽）里。因为每个 router 只管理非常少的键（最多 `O(log n / log log n)` 个），我们可以采用一种简单的二进制 Trie + 编码方法。

结构：

- 对每个待管理的键，把它哈希成一个随机二进制串（足够长）。
- 将所有键的二进制串插入一个二进制 Trie，但只存到能够与其他键区分开的最小唯一前缀。
- 在 Trie 的叶子处，存储该键对应的元数据（比如 cubby ID 和槽索引），这些值也压缩成变长比特。
- 把整棵 Trie 连同叶子值按深度优先遍历序列编码成一个紧凑的比特流。

操作：

- 查询：把查询键也哈希成二进制串，在 Trie 中沿路径走。由于 Trie 的编码是比特流，可以利用查找表一次处理多个比特（比如半个字），实现 O(1) 的跳转。
- 插入/删除：修改 Trie，更新编码比特流，同样通过查找表完成常数时间操作。

空间：由于随机二进制串的公共前缀很短，期望 Trie 的大小只与元素数量（和哈希串的随机性）有关，而不会产生巨大浪费。对于 `O(log n / log log n)` 个元素，整个编码的期望长度也就在 `O(log n)` 比特级别，且高概率不会超过 `O(log n)` 比特。

这两个组件共同保证了：无论在宏观路由还是微观元数据存储上，我们都能以几乎最优的空间实现常数时间的访问。

### 顶层：组装成最优哈希表

现在，我们要把 k-kick 树、MiniArray 和 Local Query Router 组装成一个完整的、能工作的哈希表，并且做到动态容量变化和空间信息论最优。

#### 3.1 小隔间（Cubby）的内部设计

整个哈希表被划分为许多小隔间（cubby），每个小隔间本质上就是一个独立的 k-kick 树实例，大小约为 `polylog n`。小隔间内部包含：

- 存储数组 S：长度为 `|I|`，存放键值对的实际 payload（或者商压缩后的键后缀和值）。
- 元数据 MiniArray Meta：每个槽对应一个变长条目，里面编码了三类信息：
  1.  查询路由信息（原 A_I）：用于 Local Query Router 找到槽位后的进一步定位（实际上这部分信息可能就存在路由器里，但论文把路由器放在另一个层面，此处 Meta 可能存储的是该槽位的局部路由辅助数据）。
  2.  插入元数据（原 M_I）：k-kick 树所需的控制信息，例如每个槽内元素的深度、各 bin 饱和状态等。
  3.  商压缩元数据（原 B_I）：差值 `g_I(x) - i`、部分高位比特，用于从槽位反推完整键。
- 空闲槽位树：一个 O(1) 深度的位图树，能快速找到空闲槽。

#### 3.2 宏观结构：设施（Facility）与路由

为了支持动态扩容和分摊重建成本，哈希表由 `N/K` 个设施组成。每个设施内聚了大约 `K = polylog N` 个键。

- 顶层路由 D：每个设施有一个长度为 `K` 的 MiniArray `D`。对于键 `x`，我们先通过商压缩后的哈希值 `g*(x)` 的高位选择设施，低位选择 `D` 数组中的某个位置 `b`。`D[b]` 就是一个 Local Query Router，它存着所有落到这个位置的键的目标小隔间指针和槽索引。
- 多层小隔间与 tail：为了维持高效的重建，每个设施内的小隔间有不同的大小（tier）。大部分小隔间必须是满的，只有一个称为 tail 的 1-tier 小隔间可以不满足承受插入和删除的波动。插入一律进 tail，删除非尾小隔间时从 tail 搬一个元素过来填补，以保持“除 tail 外全满”的饱和度不变量。

#### 3.3 分布不变量与后台重建

随着元素数量的变化，各 tier 小隔间的数量必须调整，以保持整体的空间利用率和元数据大小在目标范围内。通过维护一个分布不变量： 
每个设施中 tier j 的小隔间数量应大致为 $ t_j = \frac{(log^{(j)}n)^2}{(log^{(j+1))}n^2}$。

一旦某个 tier 的数量偏离目标太远（比如少于 `t_j/2` 或多于 `3t_j`），触发后台重建：

- 创建重建（数量太少）：拿一个 tier j+1 的小隔间拆分成 `t_j` 个 tier j 小隔间。
- 销毁重建（数量太多）：把 `t_j` 个 tier j 小隔间合并成一个 tier j+1 小隔间。

关键：这些重建工作被平摊到多次插入/删除操作中。例如，重建一个小隔间需要 `O(|I|)` 时间，而触发重建的频率是每 `Θ(|I|)` 次操作，因此每次操作只均摊 O(1) 的额外时间。

#### 3.4 动态扩容的整体流程（插入/查询/删除）

##### 插入(Key x)
1.  应用随机置换 π：`y = π(x)`。
2.  取 `g* = y` 的低 `log N` 位，拆分为设施索引 `r` 和路由索引 `b`。
3.  在设施 `r` 中，查询 `D[b]` 的路由器，确认键不存在。
4.  将键插入到设施 `r` 的 tail 小隔间。
    - 在 tail 内执行 k-kick 树插入，得到槽位 `slot`。
    - 在 `S[slot]` 写入键的后缀（或完整键，若未商压缩）和值。
    - 编码 `MetaEntry`（包含 delta、mid_bits 等）并写入 `M[slot]`。
    - 更新空闲槽位树。
5.  在 `D[b]` 的路由器中添加记录：键指纹 → (tail 指针, slot)。
6.  维护不变量：若 tail 变满，创建新的 tail。
7.  递增 n；若 n > N，触发扩容过程。

##### 查询(Key x)
1.  `y = π(x)`，提取 `r` 和 `b`。
2.  访问 `D[b]` 的路由器，给出小隔间 `cb` 和槽位 `slot`。
3.  从 `cb->S[slot]` 获取 payload 后缀，结合 `Meta[slot]` 中的 delta 和 mid_bits 以及 `r` 恢复出 `g*`，进而得到 `y`。
4.  应用 `π⁻¹(y)` 得到原键，验证一致性后返回值。

##### 删除(Key x)
1.  查询得到 `cb` 和 `slot`。
2.  清空 `cb->S[slot]` 和 `Meta[slot]`，更新空闲树。
3.  从路由器中移除该键。
4.  如果 `cb` 不是 tail：
    - 从 tail 中随机挑一个元素，从 tail 删除它。
    - 将该元素插入到 `cb` 刚刚空出的槽位（这不会有级联驱逐，因为空位直接可用）。
5.  如果 tail 变空，则销毁该 tail，并把另一个 1-tier 满小隔间提升为新 tail。
6.  递减 n；若 n < N/4，触发缩容。

#### 3.5 商压缩：如何达到信息论最优

上述流程中，我们依然存储了键的全部 `log|U|` 比特。商压缩的核心想法是：键的哈希值本身就是随机寻址的一部分，我们可以用寻址过程来隐式存储键的高位。

在商压缩模式下：

- 设定 `g*(x) = π(x) mod N`，也就是把键的前 `log N` 位直接用作顶层路由。
- 在小隔间中，数组 `S` 不再存储这 `log N` 位，只存储键的剩余高位和值。
- 为了在查询时恢复出完整的 `π(x)`，我们在每个槽的 `Meta` 中额外记录了修复信息：
    - `delta = g_I(x) - i`。由于 `g_I(x)` 是键在 cubby 内的随机偏好槽位，`i` 是实际存储槽位，差值恢复了 `g_I(x)`。
    - `mid_bits`：`x` 中从 `log|I|+1` 到 `log K` 的那部分比特，用来拼出完整的 `g*` 的低位。
- 有了 `g_I(x)` 和 `mid_bits`，加上已知的设施编号 `r`（提供高位），就能完整重建 `g*(x)`（即 `π(x)` 的低 `log N` 位），再拼上 `S` 中存储的高位，就得到完整的 `π(x)`，最后用 `π⁻¹` 得到真实键。

这样一来，存储数组 `S` 中每个键只占用 `log|U| - log N` 比特，而不是 `log|U|` 比特。加上路由器和 Meta 中的额外开销（均摊为 `O(log^(k) n)`），总空间接近信息论下界。

#### 3.6 总结算法全景

整个哈希表的算法逻辑可以总结为三个层次的组合：

1.  微观层（Cubby）：k-kick 树决定每个元素的实际存储位置，用 FreeSlotTree 快速找空位，所有动态信息编码进 MiniArray Meta，空间开销受控于探针复杂度。
2.  路由层（Local Query Router + D）：Trie 编码的本地路由器负责在 O(1) 时间内把查询键路由到正确的 cubby 和槽位，其空间开销与元素数量及探针复杂度匹配。
3.  宏观管理（Facility + Resize）：通过多层 cubby 结构和 tail 机制应对动态变化，通过后台重建来维护分布不变量，使得所有操作的时间保证在 `O(k)`，空间保持在 `n log(|U|/n) + O(n log^(k) n)` 比特。

```cpp
struct HashTable {
    int n;
    int N;                //  N <= n <= 2N, 其中 N 为 2 的幂值
    int K;				  // K = polylogN, 共有 N / K 个facility				
    Facility* facilities;
    ResizeManager resize;
    RebuildScheduler scheduler;
    PermutationHash pi;   // 可逆
};

struct PermutationHash {

    uint64_t k1, k2, k3, k4;
    uint64_t F(uint64_t x, uint64_t k) {
        x ^= k;
        x *= 0x9e3779b97f4a7c15;
        x ^= x >> 32;
        return x;
    }
    uint64_t pi(uint64_t x) {
        uint32_t L = x >> 32;
        uint32_t R = x & 0xffffffff;
        for (int i = 0; i < 3; i++) {
            uint32_t newL = R;
            uint32_t newR = L ^ F(R, key[i]);
            L = newL;
            R = newR;
        }
        return ((uint64_t)L << 32) | R;
    }
    uint64_t inverse(uint64_t y) {
        uint32_t L = y >> 32;
        uint32_t R = y & 0xffffffff;
        for (int i = 2; i >= 0; i--) {
            uint32_t newR = L;
            uint32_t newL = R ^ F(L, key[i]);
            L = newL;
            R = newR;
        }
        return ((uint64_t)L << 32) | R;
    }
};
```

#### Cubby

```cpp
struct Cubby {
    int tier;             		// 0,1,...,k-1
    int capacity;         		// 最大存储键数 = K / (log^(tier+1) N)^2
    int size;             		// 当前存储数量
    vector<uint64_t> S;   		// payload 数组，长度 = capacity
    MiniArray Meta;       		// 元数据，长度 = capacity，每一项编码 MetaEntry
    FreeSlotTree free_slots;
};

struct FreeSlotTree {
    int capacity;
    vector<uint64_t> bitmaps; // 从根到叶的线性化存储
    void build();             // 初始化全部为空
    int find_free();          // 返回空闲索引
    void mark_used(int i);
    void mark_free(int i);
};

int cubby_capacity(int tier) {
    int logK = (int)log2(K);
    // 计算 log^(j) N : 反复 log2 直到值≤1，这里用整数近似
    int denom = 1;
    int temp = N;
    for (int i=0; i<=tier; i++) {
        temp = (int)log2(temp);
        if (temp < 1) temp = 1;
    }
    denom = temp * temp; // (log^(tier) N)^2
    return K / denom;
}

Meta[i] = 编码以下信息：
1. query routing（原 A_I）
2. insertion metadata（原 M_I）
3. quotient metadata（原 B_I）
```

>
> 1. A[ i ] 长度不一样（0 ~ log n bit）
> 2. 还要 O(1) 访问 / 修改
> 3.  总空间还要接近信息论最优

```cpp
class MiniArray {

    PackedBTree tree;

    Entry access(int i);
    void update(int i, Entry e);
};

struct MetaEntry {

    // --- B_I ---
    int delta;          // g_I(x) - i
    uint mid_bits;      // x[log|I|+1, log K]
    // --- A_I ---
    uint router_bits;   // query routing
    // --- M_I ---
    uint insert_bits;   // k-kick / rebalance
};

Entry encode(MetaEntry m) {

    bits = []
    // delta（变长）
    bits += encode_varint(m.delta)
    // mid_bits（固定）
    bits += m.mid_bits
    // router（小）
    bits += m.router_bits
    // insert metadata（小）
    bits += m.insert_bits
    return pack(bits)
}

MetaEntry decode(Entry e) {

    MetaEntry m;
    m.delta = read_varint(e)
    m.mid_bits = read_bits(e)
    m.router_bits = read_bits(e)
    m.insert_bits = read_bits(e)

    return m
}
```



### 核心算法

#### 插入

```cpp
Insert(x):

    gx = pi(x)
    r = gx[logK+1 .. logN]
    b = gx[1 .. logK]

    F = facilities[r]
    router = F.D[b]
    // probing
    for i in 1..k:
        if free slot:
            break
    // 写 payload
    cubby.S[slot] = suffix(x)
    // 写 meta
    cubby.Meta.update(i, encode(x))
    // 更新 router
    router.insert(x, i)
    maintain_invariants(F)
```

#### 查询

```cpp
Query(x):

    gx = pi(x)
    r = gx[logK+1 .. logN]
    b = gx[1 .. logK]
    F = facilities[r]
    // router 直接返回 cubby + index
    (I, i) = F.D[b].locate(x)
    // Meta 只用于恢复
    e = I.Meta.access(i)
    return reconstruct(r, I, i, e)
```

#### 删除

```cpp
Delete(x):

    locate cubby via query path			
    remove slot
    if cubby != tail:
        move_random(tail → cubby)

    if tail empty:
        recycle_tail()
```

### 不变量维护

#### Saturation invariant

> 所有 cubby 满，除了 tail, 已在 delete / insert 中维护

#### Distribution invariant

$t_j = \frac{(log^{(j)}n)^2}{log^{(j+1)}n)^2}$  为 j 级小隔间的数量

$r_j = \frac{K}{log^{(j)}n)^2}$  为 j 级隔间的大小

#### rebuild 触发

```cpp
void maintainInvariants(Facility* F) {

    for (int j = 1; j < k; j++) {

        if (countTier(j) < t_j / 2) {
            rebuildUp(j);
        }

        if (countTier(j) > 3 * t_j) {
            rebuildDown(j);
        }
    }
}
```

#### rebuild

##### j → j+1（合并）

```cpp
void rebuildDown(int j) {

    Cubby group[t_j];

    Cubby newC = createTier(j+1);

    for each cubby in group:
        for each element:
            insertIntoCubby(newC)

    replace(group → newC)
}
```

##### j+1 → j（拆分）

```cpp
void rebuildUp(int j) {

    Cubby big = pickTier(j+1);

    Cubby small[t_j];

    randomly distribute elements

    rebuild each cubby
}
```

### Quotienting

> 可逆编码 + 索引重建

核心思想

不存完整 key，只存： 把 key = (高位 routing + 低位 payload)，但只存 payload + 少量“索引修复信息”

> x = [ facility routing bits | cubby routing bits | payload ]

重建 key

```cpp
Key reconstruct(int r, Cubby* I, int i, Entry e) {

    MetaEntry m = decode(e);
    int g_I = i + m.delta;
    uint g_K = (g_I << MID_BITS) | m.mid_bits;
    uint g_star = (r << LOG_K) | g_K;
    uint payload = I.S[i];
    return pi_inverse(g_star, payload);
}
```

### 数据库结构设计

核心表设计（逐个解释）

1️⃣ snapshot_meta（快照控制）

```
CREATE TABLE snapshot_meta (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    snapshot_tag VARCHAR(64),   -- 一次实验 / 一次保存
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    n INT,
    N INT,
    K INT
);
```

为什么要有这个？

👉 你需要：

- 标记“当前是哪一次实验”
- 支持多次运行对比
- 前端选择不同 snapshot 展示

2️⃣ facility

```
CREATE TABLE facility (
    id INT,
    snapshot_id BIGINT,

    tail_cubby_id INT,

    PRIMARY KEY (id, snapshot_id)
);
```

3️⃣ cubby（核心结构）

```
CREATE TABLE cubby (
    id INT AUTO_INCREMENT,
    snapshot_id BIGINT,

    facility_id INT,

    tier INT,
    capacity INT,
    size INT,

    is_tail BOOLEAN,

    PRIMARY KEY (id, snapshot_id)
);
```

为什么 cubby 必须存？

因为你要展示：

- tier 分布
- load factor
- tail 行为（论文重点）

👉 注意名字：snapshot，不是实时结构

为什么只存这些？

用来做什么？

为什么必须有？
