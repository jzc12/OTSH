## 一、项目概述

### 1.1 目标

复现论文 *On the Optimal Time/Space Tradeoff for Hash Tables* 中提出的哈希表结构，实现其核心性质：

- 查询：
    - 查询时间：`O(1)` 探测次数（但每次探测扫描一个大小为 $O(log^{(k)} n)$ 的 mini-bin）
    - 因此 查询时间 也是 $O(log^{(k)} n)$（如果考虑逐槽扫描），但 探测次数 是常数（仅偏好 mini-bin 和 fallback）。

- 插入：期望 O(k)，高概率成功
- 空间浪费：$O(log^{(k)} n)$ bits / key
- 支持高负载（load factor → 1）
- 功能完整：支持插入、查询、删除、动态扩容、失败重建

### 1.2 核心思想

该结构的本质为：分层随机分配（bin + mini-bin） + 有界深度重排（k-kick process）

核心要点：

- 全局划分为 bins，每个 bin 大小为 polylog(n)
- 每个 bin 内进一步划分为 mini-bins
- 每个 key 只属于一个 bin，但有一个“偏好 mini-bin”
- 插入通过最多 k 次 替换提出（k-kick）完成，保证插入时间上界
- fallback 区域保证系统始终成功（高概率）
- 查询仅搜索偏好 mini-bin + fallback，保证低查询成本

## 二、整体结构设计

### 2.1 全局结构

```text
HashTable
├─ 全局元信息（容量、负载、k值、哈希种子）
└─ B 个独立 Bin
   ├─ M 个 MiniBin（主存储区）
   └─ Fallback 数组（兜底区）
```

### 2.2 参数定义

| 参数               | 取值                                          | 说明                                  |                   |
| ------------------ | --------------------------------------------- | ------------------------------------- | ----------------- |
| n                  | 10⁴ ~ 10⁶                                     | 实际存储元素个数                      |                   |
| k                  | 1/2/3/4                                       | 最大踢出次数，控制时空 trade-off      |                   |
| bin_size           | mini_bin_size x num_mini_bins + fallback_size | 每个 bin 固定大小（多项式对数）       | 会根据 K 动态设定 |
| mini_bin_size      | $O(log^{(k)} n)$                              | 迭代对数：k=2→loglogn，k=3→logloglogn | 会根据 K 动态设定 |
| num_mini_bins      | $O(log^{(k)} n)$                              | 每个 bin 内的 mini-bin 数量           | 会根据 K 动态设定 |
| fallback_size      | $O(1)$ 设为 8                                 | 每个 bin 内的  fallback  数量         |                   |
| total_bins         | (n / bin_size)                                | 总 bin 数量                           | 会根据 K 动态设定 |
| load_factor        | 0.95 ~ 0.99                                   | 目标高负载                            |                   |
| fingerprint_bits F | 16                                            | 指纹位数，控制碰撞率                  |                   |

约束：

- 总容量 m = (1 + ε)n
- 每个 bin 内空间冗余为常数级

### 2.3 映射关系

对于 key x：

```text
hash1/hash2 完全独立
种子生成方式：随机种子

b = h_bin(x)           // 所属 bin
m = h_pref(x)          // 偏好 mini-bin
```

特点：

- 每个 key 只属于一个 bin
- 在 bin 内进行局部冲突解决

### 3.4 HashFunctions

```cpp
class HashFunc {
private:
    uint64_t seed1, seed2, seed3;
public:
    // 构造函数：初始化随机种子
    HashFunc() {
        // 生成加密安全的随机种子（避免碰撞）

        seed2 = rand() % UINT64_MAX;
        seed3 = rand() % UINT64_MAX;
    
    // 映射到 bin
    uint64_t h_bin(); // 动态扩容时会变化
    // 映射到 mini-bin
    uint64_t h_pref();
    // 计算指纹
    uint16_t fingerprint();
};
```

## 四、核心算法

### 4.1 插入算法（k-kick）

定义递归过程：

```text
Insert(x):
    1. 将 x 加入 fallback（special slot）
    2. success = place(x, 0)
    3. if success:
           从 fallback 移除 x
    4. else:
           留在 fallback

place(x, depth):
    if depth > k:
        return FAIL

    获取 x 的 bin 和 preferred mini-bin
    // 从数据库中获取

    if 有空闲:
        插入 x
        return SUCCESS
    else:
        随机选出 y
        将 x 放入 y 的位置
        将 y 加入 fallback 
        success = place(y, depth + 1)
        if success:
            从 fallback 移除 y
        return SUCCESS
```

### 4.2 查询算法

```text
1. 计算 b = h_bin(x)
2. 计算 m = h_pref(x)

3. 计算区间：
   start = b * bin_size + m * mini_bin_size
   end   = start + mini_bin_size - 1

4. 查询 mini-bin：

   SELECT fp, is_used
   FROM hash_table
   WHERE idx BETWEEN start AND end;

5. 扫描结果：
   若存在 fp 匹配 → 返回 FOUND

6. 若未找到：
   查询 fallback 区域（类似方式）

7. 返回结果
```

复杂度：$O(log^{(k)} n)$

### 4.3 删除算法

```text
b = h_bin(x)
m = h_pref(x)
在 mini-bin m 中查找，找到则删除
否则在 fallback 中查找，找到则删除
无需墓碑
```

## 八、动态扩容

```text
### 八、动态扩容（补充落地细节）
1. 触发条件：
   - 实时负载因子 = 总已用槽位 / (total_bins * bin_size) > 0.99
   - 新增“扩容中”状态标记：HashTable.is_resizing = false/true

2. 执行方式（分摊重建）：
   - 初始化扩容：
     a. 新total_bins = old_total_bins * 2
     b. 分配新的连续数组：new_table[new_total_bins * bin_size]
     c. 设置扩容进度：resize_progress = 0（表示已处理0个旧bin）
     d. 标记is_resizing = true
   - 分摊重建（每次插入/删除时执行一小部分）：
     a. 若is_resizing且resize_progress < old_total_bins：
        i. 处理第resize_progress个旧bin：遍历该bin的所有mini-bin + fallback
        ii. 对每个key x：
            - 保留h_pref(x)不变
            - 重新计算new_b = h_bin(x) % new_total_bins
            - 插入x到新表的new_b bin的h_pref(x) mini-bin
        iii. resize_progress += 1
        iv. 若resize_progress == old_total_bins：
            - 释放旧表内存
            - 切换table指针到新表
            - 更新total_bins为新值
            - 标记is_resizing = false
   - 扩容期间的读写：
     a. 查询：同时查旧表+新表（直到扩容完成）
     b. 插入：直接插入新表（保证数据一致性）

3. 代码伪模板：
class HashTable {
private:
    Entry* table;          // 当前表
    Entry* new_table;      // 扩容中的新表
    uint64_t total_bins;   // 当前bin数量
    uint64_t new_total_bins; // 新bin数量
    bool is_resizing;      // 是否扩容中
    uint64_t resize_progress; // 已处理的旧bin数量
    HashFunc hash;         // 哈希函数

public:
    // 插入时触发分摊扩容
    bool insert(uint64_t key) {
        // 第一步：分摊扩容
        if (is_resizing) {
            process_resize_step(); // 处理一个bin的迁移
        }
        // 第二步：正常插入逻辑
        // ...
    }

private:
    void process_resize_step() {
        if (resize_progress >= total_bins) {
            // 扩容完成
            delete[] table;
            table = new_table;
            total_bins = new_total_bins;
            new_table = nullptr;
            is_resizing = false;
            resize_progress = 0;
            return;
        }
        // 处理第resize_progress个旧bin
        uint64_t bin_idx = resize_progress;
        uint64_t bin_offset = bin_idx * bin_size;
        // 遍历该bin的所有槽位
        for (uint64_t slot=0; slot < bin_size; slot++) {
            uint64_t idx = bin_offset + slot;
            Entry& entry = table[idx];
            if (entry.is_used) {
                // 重新计算新bin索引
                uint64_t new_b = hash.h_bin(key, new_total_bins); // 需绑定key，这里简化
                uint64_t pref_m = hash.h_pref(key, num_mini_bins);
                // 插入新表
                insert_to_new_table(new_b, pref_m, entry);
                // 清空旧表槽位
                entry.is_used = false;
            }
        }
        resize_progress++;
    }
};
```

### 9.1 设计目标

本系统将哈希表完全存储于数据库中，不依赖内存结构进行核心计算。数据库不仅承担持久化职责，同时作为主要计算载体，直接执行插入、查询与删除操作。

设计目标如下：

- 数据库即存储 + 计算层：所有操作直接在数据库上完成
- 保持连续数组语义：逻辑结构仍为 `Entry table[]`
- 最小化内存依赖：不要求完整内存镜像
- 支持随机访问与局部扫描
- 保证结构一致性（k-kick过程）

内存中的哈希表结构为一个连续数组：

```
Entry table[total_bins * bin_size]
```

其中每个元素通过如下方式计算索引：

```
index = b * bin_size + m * mini_bin_size + slot_id
```

数据库设计需保持这一“数组语义”，即：table[i] ↔ row(idx = i)

------

### 9.3 存储方案设计

本系统提供两种数据库存储方案：



```
CREATE TABLE hash_table (
    idx BIGINT PRIMARY KEY,     -- 全局索引 index
    fp SMALLINT NOT NULL,       -- 16-bit fingerprint
    is_used BOOLEAN NOT NULL,   -- 是否占用
);
```

------


- `idx`：对应数组下标，与内存完全一致
- `fp`：存储 fingerprint（16 位）
- `is_used`：标识该 slot 是否被占用

### 9.4 初始化

系统初始化时：

```
预先插入 total_bins * bin_size 条记录
全部 is_used = false
```

这一点很重要，否则 insert 会变成 insert row
